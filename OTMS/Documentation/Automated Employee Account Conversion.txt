Automated Employee Account Conversion

Flow:

1. Applicant successfully submits the Bio-Data Onboarding Form.
2. System intercepts the successful submission confirmation.
3. System initiates the account conversion transaction.
4. System systematically migrates the applicant's data (name, application details, and secure bio-data) into a new Active Employee Entity.
5. System generates a unique Employee ID based on organizational numbering rules.
6. System triggers the "Create User Account" and "Blind Provisioning" protocols.
7. System generates a default login username and temporary password.
8. System saves the Employee Record and User Credentials to the active database.
9. System updates the original Applicant Record status to "Hired/Converted".
10. System logs the automated provisioning event.

Internal System Inputs:
- Bio-Data Form Submission Trigger (System Generated)
- Applicant Record Data (System Stored Payload):
  - First Name, Middle Name, Last Name, Suffix
  - Contact Number
  - Email Address
  - Job Position ID and Department ID
  - All 201 File fields (SSS, PhilHealth, Pag-IBIG, TIN, Bank Name, Bank Account Name/Number, Emergency Contact Name/Relationship/Number, Mother First/Middle/Last Name, Father First/Middle/Last Name)
  - Document file paths (ResumeFilePath, MedicalClearanceFilePath, NBIClearanceFilePath, PSABirthCertificateFilePath, BIRForm2316FilePath)
- ID Generation Rules (System Stored Configuration):
  - Employee Number format generated by EmployeeNumberGenerator service using organizational numbering rules
  - Default New Hire Role: "Encoder" (configurable via appsettings.json DefaultNewHireRole)

Success Messages:
- "Onboarding completed successfully. Your Employee Number is {employeeNumber}."
- Welcome email: "Your Employee Account Has Been Created" with Employee Number and Login URL
- "Employee {number} account provisioned and credentials sent to {email}." (activity log)

Error Messages:
- "Token is required."
- "Invalid or expired link."
- "This link has already been used."
- "This link has expired."
- "Default role '{role}' not found. Please contact system administrator."

System Output:
- Active Employee Record created with:
  - Employee Number (generated by organizational numbering rules)
  - Personal data migrated from ApplicantRecord (First Name, Middle Name, Last Name, Suffix, Contact Number, Email)
  - Employment Status: "Active"
  - Job Position and Department from ApplicantRecord
  - IsEmailVerified: true
- System User Account securely provisioned with:
  - Role: "Encoder"
  - Account Status: "Active"
  - IsPasswordChanged: true
  - Password hashed using ASP.NET Core PasswordHasher (blind provisioning — admin never sees the password)
- Digital 201 File completely populated on ApplicantRecord (SSS, PhilHealth, Pag-IBIG, TIN, Bank Details, Emergency Contact, Family Names, Document File Paths)
- Applicant status updated to "Hired/Converted"
- StatusHistory record added: old status -> "Hired/Converted" with remarks "Automated account conversion upon onboarding completion."
- Onboarding URL token flagged as "Used" with UsedAt timestamp
- Audit log entry generated (Action: AccountProvisioned, Target: New Employee ID — "Employee {number} account provisioned and credentials sent to {email}.")
- Audit log entry generated (Action: OnboardingCompleted — "Applicant '{name}' completed onboarding. Status updated to Hired/Converted.")
- Welcome email dispatched to applicant email via SMTP with Employee Number and login URL

Business Rules:

1. Upon the applicant's successful submission of the Onboarding form, the system shall automatically convert the applicant record into an Active Employee record.
2. This conversion shall immediately trigger User Account Creation and Blind Provisioning.
3. Blind Provisioning must ensure that the generated temporary password is not visible or exposed to the Admin.
4. The system shall execute this conversion entirely programmatically, explicitly removing the need for the Admin to manually re-encode any of the applicant's bio-data.
5. All 201 File data (government IDs, bank details, emergency contacts, family names) and document file paths from the applicant's onboarding session shall be persisted to the ApplicantRecord before conversion.
6. The Employee record shall inherit the applicant's personal information, job position, and department from the ApplicantRecord.
7. The Account shall be assigned the role specified in DefaultNewHireRole configuration (default: "Encoder").
8. Tokens shall be explicitly invalidated upon successful conversion to prevent reuse.
9. Activity logs shall be generated for both account provisioning and onboarding completion events.
